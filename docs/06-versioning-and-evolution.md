# 06-versioning-and-evolution.md

## Versioning in the repo

- **No explicit versioning scheme** is defined in the repository (no `package.json`, no CHANGELOG, no API version headers). The project is static frontend assets.
- **Cache busting**: Some CSS/JS references include a query parameter (e.g. `?v=...`). This is not generated by a build in-repo; likely applied by the serving application or manually. **TODO: Clarify with tech lead** how and when to update these.

## Migrations and history

- **No database or schema** in this repo; no migrations.
- **SCSS**: When changing variables or mixins in `_parametrs_new.scss` (or `_parametrs.scss`), recompile all entry SCSS files that import them and verify layout on landing, company, and search.
- **HTML**: Structural changes (e.g. new paywall block, changed header/footer) should be applied consistently across all three entrypoints where relevant.

## Extending without breaking existing contracts

1. **New paywall block**
   - Reuse existing `.paywall-wrapper` + `.paywall-overlay` + `.paywall-popup` markup and `paywall.scss`. Add a new content class (e.g. `.paywall-xyz`) if needed; document it in JSON-LD `cssSelector` if you add or update schema.

2. **New page**
   - Reuse header and footer structure and the same CSS link order (general → feature → paywall). Do not remove or rename IDs/classes that scripts depend on (`#header`, `#main`, `#footer`, `.paywall-wrapper`, `#dialog-content`, `#officer_items`, `#annualReport_items`).

3. **New “load more” section**
   - Use `linkMore(item)` with element attributes `page`, `child-type`, `data-recipient`; ensure the backend returns HTML that matches the existing contract (fragment with optional `hasMore`). Do not change the URL shape or response expectation without updating both frontend and backend.

4. **CSS/SCSS**
   - Prefer new classes or new partials over changing existing class names used in HTML; if renaming, update all HTML and any JS that references those classes.
   - When adding breakpoints or variables, add them to `_parametrs_new.scss` and use them in new code; avoid duplicating magic numbers.

5. **External links and forms**
   - Adding new routes (e.g. `/new-page`) or form actions requires backend support; do not point forms or critical links to unimplemented URLs. Keep links to bizprofile.net and static.bizprofile.net consistent with current usage.

## Practical checklist for safe changes

- [ ] Paywall: same wrapper/overlay/popup structure; update JSON-LD if you add or rename gated blocks.
- [ ] Header/footer: preserve IDs and main classes; keep link list and structure aligned.
- [ ] linkMore: preserve `page`, `child-type`, `data-recipient` and GET `/{childType}?page=N` contract.
- [ ] SCSS: use variables and mixins; recompile and spot-check all pages after token/breakpoint changes.
- [ ] Comments and copy: English only for new content; fix legacy Russian comments when touching those blocks.
